# 为什么你需要一条 AI 开发流水线，而不是一堆好用的工具

> 这篇文章聊的是方法论，不是产品推销。如果你已经在用 AI 写代码，并且踩过"AI 说完成了但其实没测"的坑——这篇可能值得你花十分钟。

---

## 工具够多了，流程在哪？

用 AI 写代码这件事，2025 年之后基本不是问题了。问题是：你手里的工具越来越多——spec 管理的、TDD 约束的、brainstorm 的、code review 的——但没有人告诉你它们该以什么顺序协作。

你可能装了一个 spec-driven 的框架来管理设计文档，又配了一个 TDD 行为约束来确保 AI 跑测试。但实际开发时呢？可能 TDD 模式下直接开写了，完全忘了先做 proposal；也可能 spec 写得完美，到实施阶段忘了激活验证约束。

工具物理上共存，逻辑上不协作。

这就像你有一个很好的搅拌机、一个很好的烤箱、一套很好的模具——但没有食谱。每个厨具单独都能用，但没人规定"先搅拌再入模再烘烤"的顺序时，出来的东西全凭手感。

**所以我做了一件事：不发明新工具，而是给已有的工具编排一条流水线。**

---

## 四拍流水线：Think → Grill → Do → Verify

先说结论。整条流水线就四拍：

```
Think（想清楚）→ Grill（查漏洞）→ Do（动手做）→ Verify（验收归档）
```

每一拍背后自动调度不同的能力约束：

| 拍 | 你做的事 | 背后调度的能力 |
|---|---|---|
| Think | 和 AI 做 brainstorm，生成 proposal/design/specs/tasks | 结构化提问、方案对比、artifact 管理 |
| Grill | AI 质问你的设计，找遗漏和矛盾 | 压力测试、一致性扫描、blocking/non-blocking 分级 |
| Do | 按 tasks 逐个实施 | TDD 铁律、验证证据、条件触发 subagent |
| Verify | 确认实现覆盖所有 spec，归档 | spec 逐条校验、全量测试、linter |

关键体验是：**你在每个阶段只需要敲一个命令，背后的能力调度完全透明。** 你不需要记"现在该激活哪个插件"——流水线已经编排好了。

而且每一拍之间有明确的阶段感。brainstorm 结束了，你知道进入了 artifact 生成阶段；artifacts 全部完成了，系统建议你做一轮 grill；grill 过了（或你选择跳过），才进入实施。节奏可控，不是一锅乱炖。

---

## Brainstorm：不是让 AI 自嗨，是双向对齐

很多人对 brainstorm 的理解是"让 AI 帮我想方案"。然后就一路 yes 过去了——AI 提一个方案，你说行，AI 展开细节，你说行，最后生成一堆 artifacts，你都没仔细看。

如果是这样，brainstorm 和直接让 AI 一次性生成所有文档有什么区别？**没有区别。**

brainstorm 环节真正的价值不是"生成方案"，而是**双向对齐**：

- **AI → 人**：AI 提出技术方案，你通过追问来理解它的技术细节。这是学习环节——你在搞清楚"它为什么选了这个方案而不是那个"。
- **人 → AI**：你抛出真实业务场景，让 AI 根据场景重新判断哪个方案更合理。这是校准环节——AI 不知道你的用户会怎么用、你的系统有什么历史包袱，你得告诉它。

这个形式，熟不熟悉？

就是项目评审会。以前我们团队每双周一次项目评审——项目负责人提交方案文档，评审委员会逐条质问。现在评审对象变了：以前是人写文档人评审，现在是 AI 生成方案人来评审。但形式完全一致：苏格拉底式一问一答，逐步收敛到共识。

不同的是成本。以前一次评审要凑齐五六个人的日程，现在你随时可以和 AI 做一轮。频率从双周一次变成每个变更都有。问题暴露的时机从"半个月后的评审会"变成"动手之前的十分钟对话"。

---

这里我要抛一个可能有争议的观点：

**这套流程不会降低对开发者的能力要求。恰恰相反，它对开发者的架构认知和系统理解力提出了更高的门槛。**

很多人对 AI 辅助开发的想象是"降低门槛"——让不会写代码的人也能写代码，让初级开发者产出高级的设计。但这套流程的底色完全不是这个方向。它的底色是**工程决策能力的放大器，而非替代品**。

想清楚这件事：brainstorm 阶段 AI 提出三个架构方案让你选——你得能判断哪个方案在你的业务场景下更合理。Grill 阶段 AI 质问你的 error path 覆盖——你得能理解为什么这个边界条件在生产环境会炸。TDD 阶段 AI 帮你生成测试——你得能识别哪些测试是在验证业务不变量，哪些只是在测实现细节。

**每一个环节，AI 都在问你问题。但如果你对系统没有整体认知，你连问题本身都无法正确回答。**

这和传统工具链的逻辑完全不同。传统工具降低的是"执行成本"——自动补全让你打字更快，linter 让你犯更少的格式错误。但这套流程降低的是"决策摩擦"——它让架构决策的发生频率更高、反馈周期更短。而决策频率越高，对决策者的认知水平要求就越高，不是越低。

所以我的判断是：AI 工作流最终会拉大"能做架构决策的开发者"和"只能执行既定方案的开发者"之间的差距。流程能保证你不遗漏、不跑偏，但它无法替你建立对系统的整体判断力。

## Grill：出厂前的质检

brainstorm 是生产环节的双向对齐。但人终究会疲惫——尤其当你已经和 AI 聊了二十分钟、脑子里全是实现细节的时候，你很可能无意识地一路 yes 过去了。

Grill 就是为这个场景设计的兜底：**所有 artifacts 生成完毕后，AI 反过来质问你的设计。**

注意这里角色关系的变化：
- brainstorm 阶段：AI 提问挖掘你的场景知识 → 生产
- grill 阶段：AI 质问已生成的方案 → 质检

Grill 会从多个维度审查你的 artifacts——范围边界清不清晰、错误路径有没有覆盖、依赖引入合不合理、能不能测试、有没有性能陷阱。每次只问一个问题，标注 blocking（必须改）或 non-blocking（建议改），并且给出修改预览。你确认后才写入。

**这个环节是建议执行，不是强制的。** 如果你对 brainstorm 的结果有信心，可以直接跳过进入实施。但对于大变更、或者你确实一路 yes 过来的场景——跑一轮 grill 的成本是几分钟，收益是在写代码之前把设计缺陷堵住。此时修改的成本是改几个文本文件，而不是回滚几百行已经写好的代码和测试。

## TDD：在 AI 场景下，它的意义变了

TDD 在传统开发中的核心价值是"驱动设计"——通过先写测试来思考接口。

但在 AI 编码场景下，TDD 的核心价值变成了另一件事：**提供可验证的完成证据。**

AI 说"这个任务完成了"——你信吗？如果它声称跑过测试了——你怎么验证？

答案是：**要求它粘贴终端原始输出。** 不接受"我认为它能工作"，不接受"逻辑上应该没问题"，只接受"这是我刚跑的 vitest 输出，12 个测试全绿"。

这就是为什么在流水线的 Do 阶段，TDD 不是建议而是铁律：写失败测试 → 最小实现 → 粘贴测试通过的证据 → 才能标记完成。不是因为 TDD 是什么信仰，而是因为在 AI 编码场景下，**测试是你唯一能观察到的、AI 确实做了验证的证明。**

如果测试是可选的，AI 在上下文压力大的时候会跳过验证——这是观察到的实际行为，不是猜测。

## 编排的意义：为什么不是"装三个插件"

你可能会想：我自己装一个 spec 管理工具、配一个 TDD 约束、再加一个 review 流程，不也一样？

技术上一样，体验上完全不同。

区别在于：**谁来记住"现在该激活什么"？**

如果是你来记——你会忘。尤其当你第三个小需求的时候，你会想"这个太简单了不需要 brainstorm"，然后写到一半发现方向错了。

如果是流水线来记——每个命令背后自动调度对应的能力。你调用"开始实施"，TDD 约束和验证要求就自动激活了，不需要你手动开启。

这就是"融合"和"组合"的区别：组合是"你有三个好工具"，融合是"你有一条流水线，工具在正确的时刻自动接力"。

技术思路上，这条流水线是通过声明式规则实现的——把"什么阶段激活什么能力、以什么约束强度执行"写成规则。规则有分级：绝对不可违反的（比如 TDD 铁律）标为 MUST，通常正确但有例外的（比如"6 个以上独立任务时建议启用并行模式"）标为 SHOULD 并附带触发条件。

这不是理想方案——理想方案是系统级强制执行。但在当前技术条件下，声明式规则是成本最低、效果最好的选择。

## 试试看

如果上面的思路让你有共鸣（或者有强烈反对意见——也欢迎），可以体验一下 SpecFuse：

```bash
npm create specfuse@latest my-project
```

一条命令，把上面说的整条流水线注入到你的项目里。brainstorm、grill、TDD 铁律、artifact 管理——全部开箱可用。

不想用工具也没关系。这篇文章的核心观点是方法论层面的：**AI 编码需要的不是更多好用的工具，而是一条把工具编排成连贯节奏的流水线。** 你完全可以用自己的方式实践这个思路。
