# SpecFuse 产品概览

> 将 AI 氛围编码改造为工业级流水线：**OpenSpec 定方向，Superpowers 定纪律。**

**版本：** v0.2.0 | **日期：** 2026-05-25

---

## 一、行业背景与产品定位

### AI 编码的现状与问题

2025-2026 年，AI 编码助手经历了从"补全代码片段"到"独立完成功能"的跃迁。Claude Code、GitHub Copilot Workspace、Cursor Agent 等产品让开发者可以用自然语言描述需求，AI 自动生成完整实现。

然而，**能写代码 ≠ 能写好代码。**

行业观察到的普遍问题：

| 现象 | 根因 | 后果 |
|------|------|------|
| AI 生成代码后声称"已完成"但未跑测试 | 没有强制验证机制 | Bug 到集成阶段才暴露，修复成本翻倍 |
| 复杂 feature 直接开写，缺少设计 | 没有结构化的需求推导过程 | 方向错误导致整体返工 |
| 不同开发者的 AI 产出风格迥异 | 没有统一的行为约束 | Code review 负担加重，技术债积累 |
| 大变更写到一半 AI 开始"遗忘"前文 | 上下文窗口有限，无隔离手段 | 后半段代码质量显著低于前半段 |
| 团队无法追溯"为什么这样实现" | AI 对话是临时的，决策过程不持久化 | 新人接手困难，重复犯同样的错误 |

**核心洞察：** AI 编码工具的瓶颈已经不是"能力"，而是"纪律"。模型能力在快速进步，但工程流程约束仍然是手动配置、因人而异的。

### SpecFuse 是什么

SpecFuse 是一个**工程方法论即代码（Methodology-as-Code）**产品。它将经过验证的软件工程最佳实践——TDD、spec-driven development、structured code review——编码为机器可执行的约束规则，通过一条命令注入到任何项目中。

```
传统方式：工程师记住规范 → 手动遵守 → 人工审查
SpecFuse：规范编码为 CLAUDE.md → AI 自动遵守 → 流程强制验证
```

它不是另一个 AI 编码工具，而是一个让现有 AI 编码工具**按规矩办事**的约束层。

### 产品愿景

> **让每个使用 AI 编码的项目，都能达到高级工程师 code review 能通过的标准，无论使用者本身的经验水平如何。**

## 二、目标用户与使用场景

### 目标用户画像

| 用户类型 | 痛点 | SpecFuse 价值 |
|----------|------|---------------|
| **技术 Lead / 架构师** | 团队用 AI 写代码但质量不可控，review 负担重 | 一次配置、全员统一，AI 行为可预测 |
| **高级工程师** | 知道 TDD 好但 AI 总跳过，手动提醒低效 | 规则写进项目，不再需要每次口头提醒 AI |
| **中级工程师** | 想用 AI 加速但不确定产出质量是否达标 | 流程保证底线，放心让 AI 接管更多工作 |
| **初创团队 CTO** | 团队小、迭代快，无暇制定完整工程规范 | 30 秒获得工业级流程，开箱即用 |
| **开源项目维护者** | 接受 AI 生成的 PR 但质量参差不齐 | 贡献者的 AI 自动遵守项目规范 |

### 核心使用场景

**场景 1：新项目冷启动**

> "我要开一个新的 Rust 微服务项目，希望从第一行代码开始就有完整的工程规范。"

```bash
create-specfuse payment-service --stack rust
```

30 秒内获得：项目骨架 + TDD 约束 + spec 工作流 + 权限白名单。从第一个 feature 开始就是"设计 → 测试 → 实现 → 验证"的正循环。

**场景 2：已有项目规范化改造**

> "现有 Go 项目已经跑了半年，想引入结构化的 AI 开发流程但不影响现有代码。"

```bash
cd existing-service && create-specfuse --stack go
```

智能合并：只注入方法论约束，不碰现有代码、配置和 CI。已有的 `.gitignore`、权限设置只追加不覆盖。

**场景 3：大规模 feature 开发**

> "这个 feature 涉及 8 个模块改动，每次 AI 写到后面就开始出错。"

当 tasks.md 中任务数 ≥ 6 且任务独立时，SpecFuse 生成的约束会引导 AI 自动切换到 subagent 模式：每个模块一个独立 subagent，各自 TDD，主 agent 做两阶段审查后合并。上下文不再是瓶颈。

**场景 4：团队规范统一**

> "5 个人用 Claude Code，每个人的 AI 行为都不一样，产出质量无法预期。"

所有项目用同一个 SpecFuse 配置初始化 → 同一份 CLAUDE.md → 所有人的 AI 助手行为一致。技术 Lead 只需维护一份栈模板，团队自动继承。

**场景 5：探索性开发的安全网**

> "我想试一个新方案，但怕搞坏现有代码。"

`/opsx:explore` + worktree 隔离 → 在独立分支上自由实验，结论满意则合并，不满意直接丢弃，零成本。

### 技术选型

| 层 | 技术 | 理由 |
|---|---|---|
| CLI 框架 | Commander + Inquirer | 成熟、零配置、支持交互/非交互双模式 |
| 类型校验 | Zod | 运行时校验自定义栈 YAML 的结构正确性 |
| 构建 | TypeScript + ESM | 类型安全，原生模块系统 |
| 测试 | Vitest | 快速、兼容 ESM、零配置 |
| 分发 | npm + git | npm 面向公开生态，git 面向私有部署 |

### 依赖关系

```
create-specfuse (脚手架，安装时使用)
    │
    ├── @fission-ai/openspec (运行时，spec 工作流引擎)
    │
    └── obra/superpowers (运行时，Claude Code 行为约束插件)
```

三者独立发布、独立升级。SpecFuse 通过 CLAUDE.md 桥接两者，不修改上游源码。

## 五、整合流程：三阶段工程流水线

### 整体流程图

```
用户想法
    │
    ▼
┌─────────────────────────────────────────────┐
│  阶段 1：Think（方案设计）                    │
│  /opsx:explore → /opsx:propose 或 /opsx:new │
│                                              │
│  ┌─ brainstorming ─────────────────────┐     │
│  │  苏格拉底式一次一问                    │     │
│  │  2-3 种方案对比 + 推荐              │     │
│  │  Non-goals + Trade-offs             │     │
│  └─────────────────────────────────────┘     │
│                                              │
│  产出：proposal → design → specs → tasks    │
└──────────────────────┬──────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────┐
│  阶段 2：Do（实施执行）                      │
│  /opsx:apply                                │
│                                              │
│  ┌─ TDD 铁律 ──────────────────────────┐     │
│  │  RED: 写失败测试                      │     │
│  │  GREEN: 最小实现                     │     │
│  │  REFACTOR: 保持绿灯重构              │     │
│  └─────────────────────────────────────┘     │
│                                              │
│  ┌─ 条件升级 ──────────────────────────┐     │
│  │  ≥6 任务 + 独立 → Subagent 分派     │     │
│  │  破坏性变更 → Worktree 隔离          │     │
│  └─────────────────────────────────────┘     │
└──────────────────────┬──────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────┐
│  阶段 3：Verify（验收归档）                  │
│  /opsx:verify → /opsx:archive               │
│                                              │
│  ✓ 实现 vs spec 逐条对照                     │
│  ✓ 全量测试通过                              │
│  ✓ Linter 零警告                             │
│  ✓ 归档变更，进入历史记录                     │
└─────────────────────────────────────────────┘
                       │
                       ▼
                  可交付代码
```

### 阶段 1：方案设计（Think）

```
触发：/opsx:explore → /opsx:propose 或 /opsx:new
约束：Superpowers brainstorming 技能
```

| 强制行为 | 目的 |
|----------|------|
| 苏格拉底式一次一问 | 防止 AI 一次性抛出大量问题导致用户认知过载 |
| 2-3 种方案对比 | 防止 AI 直奔第一个想到的方案，忽略替代选项 |
| Non-goals + Trade-offs | 明确边界，记录决策理由，便于未来追溯 |
| 产出 artifact 序列 | proposal → design → specs → tasks，结构化推导 |

**产品意义：** 这个阶段确保了 AI 不是"接到需求就开写"，而是先充分理解问题、考虑替代方案、与人对齐后才动手。减少了最昂贵的错误——方向性错误。

### 阶段 2：实施执行（Do）

```
触发：/opsx:apply
约束：TDD + verification-before-completion + subagent/worktree（条件触发）
```

| 强制行为 | 目的 |
|----------|------|
| 先写失败测试再实现 | 确保每个功能都有覆盖，测试证明了它能捕获缺陷 |
| 跑测试贴原始输出 | 杜绝"我相信它能过"式的虚假完成声明 |
| Subagent 分派（≥6 任务） | 大变更时隔离上下文，防止会话过长导致质量下降 |
| Worktree 隔离（破坏性变更） | 保护主分支，失败可零成本丢弃 |

**产品意义：** 这个阶段解决了 AI 编码最常见的两个问题：(1) 声称完成但实际没测试；(2) 大变更后半段质量下滑。通过铁律约束和智能扩缩，保证无论变更大小都有可靠的质量基线。

### 阶段 3：验收归档（Verify）

```
触发：/opsx:verify → /opsx:archive
约束：全量测试 + linter 零警告 + spec 对照
```

| 强制行为 | 目的 |
|----------|------|
| 实现与 spec 逐条对照 | 防止实现偏离设计，遗漏需求 |
| 全量测试通过 | 确认无回归 |
| Linter 零警告 | 维护代码质量基线 |

**产品意义：** 传统开发中，"做完了"和"验收通过"之间往往有 gap。这个阶段填补了这个 gap——AI 不能自己说"完成了"，必须拿出证据。

## 六、为什么这么整合

### 行业认知：两个半成品的融合

OpenSpec 和 Superpowers 各自解决了 AI 编码的一半问题：

| 维度 | 仅 OpenSpec | 仅 Superpowers | SpecFuse 融合 |
|------|-------------|----------------|--------------|
| **方向感** | ✅ 有 spec 驱动的工作流 | ❌ 只有执行技能，不管做什么 | ✅ OpenSpec 定方向 |
| **执行纪律** | ❌ 不强制 TDD，不验证完成 | ✅ 严格的行为约束 | ✅ Superpowers 定纪律 |
| **变更生命周期** | ✅ proposal→archive 完整记录 | ❌ 无持久化 | ✅ 完整追溯链 |
| **自适应扩缩** | ❌ 无论大小变更流程相同 | ✅ 有 subagent/worktree | ✅ 小变更轻量，大变更自动升级 |
| **多栈适配** | ❌ 手写配置 | ❌ 手写配置 | ✅ 一条命令自动适配 |

单独使用任何一个，都会有明显短板。SpecFuse 的价值在于：**让两者各司其职、协同工作，且用户无需了解底层细节。**

### 融合的关键设计决策

**决策 1：CLAUDE.md 作为"桥接层"**

不修改 OpenSpec 或 Superpowers 的源码，而是通过 CLAUDE.md 声明两者的协作规则。这意味着：
- 两个上游项目可以独立升级，不担心兼容性
- 融合规则可以按项目定制，不同项目可以有不同的约束强度
- 无 vendor lock-in，随时可以移除 SpecFuse 只保留其中之一

**类比：** 就像 Docker Compose 不修改任何容器镜像，只定义它们如何协作。

**决策 2：声明式触发条件替代硬编码逻辑**

Subagent 和 Worktree 的启用不是代码逻辑判断，而是写在 CLAUDE.md 中的声明式规则。AI 根据当前状态（任务数、依赖关系、上下文压力）自行判断是否满足条件。

**为什么不硬编码？** 因为"何时需要 subagent"是一个需要上下文判断的决策。同样是 6 个任务，如果它们紧密耦合就不该用 subagent；如果它们完全独立就应该用。声明式规则给了 AI 判断空间，同时约束了判断框架。

**决策 3：渐进式复杂度**

```
小变更（1-3 任务）  → 直接 TDD，无额外开销
中等变更（4-5 任务） → TDD + 可选 worktree
大变更（≥6 任务）   → TDD + subagent + worktree（自动建议）
```

**为什么分级？** 因为工程流程的成本必须与变更的复杂度匹配。给一个单行 bugfix 跑 subagent 是浪费；给一个 8 模块 feature 不做隔离是冒险。SpecFuse 让流程成本自适应。

## 七、整合后的优势

### 量化收益预期

| 指标 | 裸用 AI 编码 | SpecFuse 融合流程 | 改善 |
|------|-------------|------------------|------|
| 测试覆盖率 | 依赖开发者提醒，通常 40-60% | 100% 强制（TDD 铁律） | +40-60% |
| 设计文档产出 | 通常无 | 每个变更自动产出 | 从 0 到 100% |
| 方向性返工率 | 高（缺少方案对比） | 低（强制 Non-goals + 替代方案） | 预计降低 50%+ |
| 大变更后半段质量 | 显著下降（上下文饱和） | 保持稳定（subagent 隔离） | 质量一致性 |
| 新人上手时间 | 需要学习团队规范 | 一条命令落地，AI 自动遵守 | 预计缩短 70% |

### 对个人开发者

| 优势 | 具体表现 |
|------|----------|
| **质量保底** | 即使赶工期，AI 也不能跳过测试和验证——规则写死在 CLAUDE.md 中 |
| **减少返工** | 方案阶段强制考虑替代方案和非目标，减少"做完发现方向错"的情况 |
| **大变更可控** | Subagent + Worktree 让 8+ 任务的大 feature 不再失控 |
| **知识沉淀** | 每个变更的 proposal/design/specs 自动留下决策记录 |
| **信心提升** | 有验证证据支撑的"完成"，不再是"我觉得应该没问题" |

### 对团队

| 优势 | 具体表现 |
|------|----------|
| **一致性** | 所有人的 AI 助手遵循相同流程，代码风格和质量标准统一 |
| **可审计** | 每个变更有完整的 artifact 链（为什么做 → 怎么设计 → 怎么验证） |
| **新人友好** | `create-specfuse` 一条命令即完成项目规范落地，新人无需学习配置 |
| **CI 集成** | `--yes` 模式支持自动化初始化，可嵌入项目模板 |
| **Review 减负** | AI 产出已经过 TDD + 两阶段审查，人工 review 聚焦架构而非琐碎 |

### 对组织

| 优势 | 具体表现 |
|------|----------|
| **规范落地零摩擦** | 不再是"规范文档写了但没人看"，而是规范直接控制 AI 行为 |
| **合规友好** | 每个变更有完整审计链，满足 ISO/SOC2 等对开发过程可追溯的要求 |
| **风险可控** | 破坏性变更强制隔离，大变更有结构化分解，降低线上事故概率 |

## 八、当前设计的不足

### 已知局限

| 局限 | 影响 | 严重度 | 缓解方向 |
|------|------|--------|----------|
| **Subagent 触发是建议性的（SHOULD）** | AI 可能偶尔跳过 subagent 模式 | 中 | 未来升级为 MUST + tasks.md 自动计数检测 |
| **上下文消耗判断不精确** | "已消耗 >40%" 依赖 AI 自估 | 低 | 等待 Claude Code 暴露精确的 context usage API |
| **Subagent 串行执行** | 即使任务独立，implementer 仍需逐个执行 | 中 | Superpowers 上游限制，持续跟进 |
| **仅支持 Claude Code** | 依赖 CLAUDE.md 约定和 Superpowers 插件体系 | 高 | v0.3 路线图：Gemini CLI + Copilot 适配 |
| **无运行时强制** | 声明式规则，AI 是否真正执行无系统级保障 | 中 | 可增加 hook 验证（如 pre-commit 检查测试覆盖率） |
| **自定义栈无约束继承** | 自定义栈只能定义命令，不能定义栈特有的工作流规则 | 低 | 未来支持 `rules:` override in YAML |

### 架构债务

| 项目 | 说明 | 优先级 |
|------|------|--------|
| `dist/` 提交到仓库 | 为支持 git-based npm install 不得不提交编译产物 | 低（发布到 npm 后可移除） |
| 模板函数无独立单测 | 当前测试粒度是整体输出，单个 render 函数未独立测试 | 中 |
| 无 E2E 验证 | 缺少"生成项目→启动 Claude Code→验证 TDD 激活"的自动化 | 高 |
| 无版本兼容性矩阵 | 未定义与 OpenSpec / Superpowers 各版本的兼容关系 | 中 |

## 十、总结

### 核心价值主张

> **SpecFuse 不是一个更好的 AI 模型，而是一套让任何 AI 模型都能按工程标准工作的约束框架。**

它回答的问题是：**在 AI 能力已经足够的今天，如何确保这些能力被正确使用？**

答案是三个层次的保障：

1. **流程保障** — 三阶段流水线确保"想清楚再动手，做完要验证"
2. **纪律保障** — TDD 铁律 + 验证证据确保每一步都有据可查
3. **规模保障** — Subagent + Worktree 确保大变更不失控

### 当前状态（v0.2.0）

- ✅ 四栈支持（Rust / Go / TypeScript / Python）+ 自定义栈
- ✅ 三阶段完整流水线（Think → Do → Verify）
- ✅ 条件触发的 Subagent + Worktree 自动升级
- ✅ 绿地创建 + 已有项目智能合并
- ✅ 110 个测试全部通过

### 下一步

多 AI 平台适配 → 自动检测升级 → Plugin 生态 → 团队级配置管理。
