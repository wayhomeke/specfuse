# SpecFuse 产品概览

> 将 AI 氛围编码改造为工业级流水线：**OpenSpec 定方向，Superpowers 定纪律。**

---

## 一、产品定位

SpecFuse 是一个面向 AI 辅助开发场景的**工程方法论脚手架**。它解决的核心问题是：

**当 AI 编程助手（如 Claude Code）拥有了写代码的能力，谁来保证它写出的是正确的、可维护的、符合工程标准的代码？**

答案不是更好的模型，而是更好的流程约束。SpecFuse 将两个互补的开源工具——OpenSpec（spec 驱动工作流引擎）和 Superpowers（AI 行为约束插件）——融合为一套端到端的工程流水线，通过一条命令注入到任何项目中。

| 层 | 技术 | 理由 |

| 分发 | npm + git | npm 面向公开生态，git 面向私有部署 |

## 三、项目目标

### 解决的问题

1. **AI 编码缺乏工程纪律** — 模型会跳过测试、忘记验证、过度实现、遗漏边界条件
2. **方法论落地成本高** — TDD、spec-driven、code review 都是好实践，但手动配置繁琐
3. **团队一致性差** — 不同开发者的 AI 助手行为不一致，代码质量参差不齐
4. **大变更缺乏结构** — 复杂 feature 直接开写，缺少设计推导和任务拆解

### 设计原则

- **一条命令注入** — `create-specfuse my-app --stack rust`，30 秒完成
- **声明式约束** — 通过 CLAUDE.md 声明规则，AI 自动遵守，无需人工监督
- **渐进式升级** — 小变更用默认流程，大变更自动触发 subagent + worktree
- **栈无关** — 方法论与具体技术栈解耦，同一套流程适用于 Rust/Go/TS/Python

## 四、整合流程：三阶段工程流水线

SpecFuse 将软件开发划分为三个阶段，每个阶段绑定对应的 AI 行为约束：

### 阶段 1：方案设计（Think）

```
触发：/opsx:explore → /opsx:propose 或 /opsx:new
约束：Superpowers brainstorming 技能
```

| 强制行为 | 目的 |
|----------|------|
| 苏格拉底式一次一问 | 防止 AI 一次性抛出大量问题导致用户认知过载 |
| 2-3 种方案对比 | 防止 AI 直奔第一个想到的方案，忽略替代选项 |
| Non-goals + Trade-offs | 明确边界，记录决策理由，便于未来追溯 |
| 产出 artifact 序列 | proposal → design → specs → tasks，结构化推导 |

### 阶段 2：实施执行（Do）

```
触发：/opsx:apply
约束：TDD + verification-before-completion + subagent/worktree（条件触发）
```

| 强制行为 | 目的 |
|----------|------|
| 先写失败测试再实现 | 确保每个功能都有覆盖，测试证明了它能捕获缺陷 |
| 跑测试贴原始输出 | 杜绝"我相信它能过"式的虚假完成声明 |
| Subagent 分派（≥6 任务） | 大变更时隔离上下文，防止会话过长导致质量下降 |
| Worktree 隔离（破坏性变更） | 保护主分支，失败可零成本丢弃 |

### 阶段 3：验收归档（Verify）

```
触发：/opsx:verify → /opsx:archive
约束：全量测试 + linter 零警告 + spec 对照
```

| 强制行为 | 目的 |
|----------|------|
| 实现与 spec 逐条对照 | 防止实现偏离设计，遗漏需求 |
| 全量测试通过 | 确认无回归 |
| Linter 零警告 | 维护代码质量基线 |

## 五、为什么这么整合

### OpenSpec 和 Superpowers 的互补关系

| 维度 | 仅 OpenSpec | 仅 Superpowers | SpecFuse 融合 |
|------|-------------|----------------|--------------|
| **方向感** | ✅ 有 spec 驱动的工作流 | ❌ 只有执行技能，不管做什么 | ✅ OpenSpec 定方向 |
| **执行纪律** | ❌ 不强制 TDD，不验证完成 | ✅ 严格的行为约束 | ✅ Superpowers 定纪律 |
| **变更生命周期** | ✅ proposal→archive 完整记录 | ❌ 无持久化 | ✅ 完整追溯链 |
| **自适应扩缩** | ❌ 无论大小变更流程相同 | ✅ 有 subagent/worktree | ✅ 小变更轻量，大变更自动升级 |
| **多栈适配** | ❌ 手写配置 | ❌ 手写配置 | ✅ 一条命令自动适配 |

### 融合的关键设计决策

**决策 1：CLAUDE.md 作为"桥接层"**

不修改 OpenSpec 或 Superpowers 的源码，而是通过 CLAUDE.md 声明两者的协作规则。这意味着：
- 两个上游项目可以独立升级
- 融合规则可以按项目定制
- 无 vendor lock-in

**决策 2：声明式触发条件替代硬编码逻辑**

Subagent 和 Worktree 的启用不是代码逻辑判断，而是写在 CLAUDE.md 中的声明式规则。AI 根据当前状态（任务数、依赖关系、上下文压力）自行判断是否满足条件。这比硬编码阈值更灵活，因为 AI 可以结合上下文做判断。

**决策 3：渐进式复杂度**

```
小变更（1-3 任务）  → 直接 TDD，无额外开销
中等变更（4-5 任务） → TDD + 可选 worktree
大变更（≥6 任务）   → TDD + subagent + worktree（自动建议）
```

不给简单任务增加仪式感开销，只在复杂度真正需要时才引入更重的工具。

## 六、整合后的优势

### 对个人开发者

| 优势 | 具体表现 |
|------|----------|
| **质量保底** | 即使赶工期，AI 也不能跳过测试和验证——规则写死在 CLAUDE.md 中 |
| **减少返工** | 方案阶段强制考虑替代方案和非目标，减少"做完发现方向错"的情况 |
| **大变更可控** | Subagent + Worktree 让 8+ 任务的大 feature 不再失控 |
| **知识沉淀** | 每个变更的 proposal/design/specs 自动留下决策记录 |

### 对团队

| 优势 | 具体表现 |
|------|----------|
| **一致性** | 所有人的 AI 助手遵循相同流程，代码风格和质量标准统一 |
| **可审计** | 每个变更有完整的 artifact 链（为什么做 → 怎么设计 → 怎么验证） |
| **新人友好** | `create-specfuse` 一条命令即完成项目规范落地，新人无需学习配置 |
| **CI 集成** | `--yes` 模式支持自动化初始化 |

### 相比"裸用" AI 编码

| 指标 | 裸用 Claude Code | SpecFuse 融合流程 |
|------|-----------------|------------------|
| 测试覆盖率 | 依赖开发者提醒 | 100% 强制（TDD 铁律） |
| 设计文档 | 通常无 | 每个变更自动产出 |
| 回退成本 | 高（代码耦合在主分支） | 低（worktree 隔离） |
| 大变更质量 | 上下文过长后显著下降 | subagent 隔离上下文 |
| 完成可信度 | "我觉得应该能过" | 必须贴终端原始输出 |

## 七、当前设计的不足

### 已知局限

| 局限 | 影响 | 缓解方式 |
|------|------|----------|
| **Subagent 触发是建议性的** | AI 可能不严格遵守 SHOULD 级别的规则，偶尔跳过 subagent 模式 | 未来可升级为 MUST + 自动检测 |
| **上下文消耗判断不精确** | "会话上下文已消耗 >40%" 依赖 AI 自估，无精确 API | 等待 Claude Code 暴露精确的 context usage 指标 |
| **Subagent 不能并行实现** | 即使任务独立，implementer subagent 仍需串行执行，只能并行审查 | Superpowers 上游限制，非 SpecFuse 可控 |
| **仅支持 Claude Code** | 当前模板基于 Claude Code 的 CLAUDE.md 约定和 Superpowers 插件体系 | 未来可扩展 Gemini CLI (GEMINI.md) 和 Copilot 支持 |
| **无运行时验证** | 触发条件是声明式规则，AI 是否真正执行无法在系统层面强制 | 依赖用户在关键节点审查 AI 行为 |
| **自定义栈无约束继承** | 用户自定义的栈只提供命令和模式，无法定义栈特有的工作流规则 | 未来可支持栈级别的 rule override |

### 架构债务

| 项目 | 说明 |
|------|------|
| `dist/` 提交到仓库 | 为支持 git-based npm install 不得不提交编译产物，增加 diff 噪音 |
| 模板函数无独立单测 | 当前测试粒度是 `composeCLAUDEmd` 整体输出，单个 render 函数未独立测试 |
| 无 E2E 验证 | 缺少"生成项目→在其中启动 Claude Code→验证 TDD 被激活"的端到端自动化测试 |

### 路线图方向

1. **自动检测升级** — 当 tasks.md 满足 subagent 条件时，自动提示用户确认启用
2. **多 AI 平台支持** — 生成 GEMINI.md / AGENTS.md 适配其他 AI 编码工具
3. **Plugin 生态** — 支持社区贡献栈模板和自定义工作流 schema
4. **Metrics 仪表盘** — 收集 TDD 通过率、变更完成时间、回退率等指标
5. **团队配置继承** — 组织级 SpecFuse 配置，新项目自动继承团队规范

```bash
git clone ssh://git@gitlab.fatehome.net:30021/fatehome/specfuse.git
cd specfuse && npm install && npm run build && npm link

# 创建项目

claude                    # 启动 Claude Code

/opsx:verify             # 验收

```

## 九、总结

SpecFuse 的核心价值主张：

> **不要相信 AI 的自觉性，要相信流程的强制性。**

它不是一个更好的 AI 模型，而是一套让任何 AI 模型都能按工程标准工作的约束框架。通过将 OpenSpec 的工作流管理能力和 Superpowers 的行为约束能力融合为一条自动化流水线，SpecFuse 让"AI 写的代码"达到了"人类工程师 code review 能通过"的标准。

当前版本（v0.2.0）已实现：
- 四栈支持（Rust / Go / TypeScript / Python）+ 自定义栈
- 三阶段完整流水线（Think → Do → Verify）
- 条件触发的 Subagent + Worktree 自动升级
- 绿地创建 + 已有项目智能合并

下一步重点：多 AI 平台适配、自动检测升级、E2E 测试覆盖。
