# SpecFuse 设计手记

> 这不是产品手册，也不是架构文档。这是一份关于"为什么要做这件事、怎么想的、怎么做的"的设计笔记。记录思考过程比记录结论更有价值——因为结论会过时，但思考方式可以复用。

---

## 一、意图：我们到底在解决什么问题

### 问题不是 AI 写不好代码

2025 年之后，"AI 能不能写代码"已经不是问题了。Claude Code 能从零搭建一个完整的微服务，Cursor 能一次性重构一个组件树。能力层面，AI 已经是一个合格的中高级开发者。

**真正的问题是：AI 的产出质量不稳定，而且你无法预测它什么时候会出问题。**

这不是能力问题，是一致性问题。同一个 AI，同样的指令，今天写出来是 TDD 全覆盖的干净代码，明天就可能跳过测试、遗漏边界、声称"已完成"但实际没跑过一次 `vitest run`。你不知道哪一次交付是靠谱的——除非你每次都亲自验证，但那就失去了用 AI 的意义。

### 痛点的本质：信任缺口

把问题再抽象一层：开发者和 AI 之间存在一个**信任缺口**。

- AI 说"done"，你不敢直接 merge
- AI 说"测试通过"，你不确定它真的跑了
- AI 在第 3 个任务时表现完美，到第 7 个任务时开始犯低级错误——但它自己不会告诉你
- 你想让 AI 做更多，但怕放手后兜不住

这个缺口的存在，让 AI 编码的实际效率远低于理论效率。开发者要么花大量时间审查 AI 产出（低效），要么盲目信任然后在集成阶段爆炸（高风险）。

**SpecFuse 的意图就是填补这个信任缺口：让你在合理的成本下，相信 AI 的产出是可靠的。**

不是通过"更好的 AI"来解决——模型能力进步了但一致性问题仍然在。而是通过流程约束：你不需要每次都亲自验证 AI 有没有跑测试，因为流程规则让它不跑测试就无法标记完成。

## 二、痛点拆解：四个具体的工程问题

把"信任缺口"拆开，落到日常开发场景中，是四个具体的痛点：

### 痛点 1：完成声明不可信

AI 标记任务完成，但它对"完成"的定义和你不同。你认为"完成"意味着代码能工作、测试能通过、边界被覆盖。AI 认为"完成"意味着代码写好了、逻辑上应该没问题。

中间差的那一步——**实际执行验证命令并确认输出**——AI 经常跳过。不是故意的，是因为在没有约束的情况下，"写完代码"和"验证代码"之间没有硬性关联。

### 痛点 2：大变更后半段质量塌方

Transformer 的上下文窗口是有限的。当一个变更涉及 8 个任务、改动 12 个文件时，AI 在处理第 7、8 个任务时，对前面任务的"记忆"已经开始模糊。具体表现：

- 命名不一致（前面用 `userId`，后面用 `user_id`）
- 重复实现已有的 helper 函数
- 忘记前面定义的接口约定
- 错误处理风格前后不统一

这不是 AI "变笨了"，是 attention 机制的物理限制。但效果是一样的：后半段代码的质量显著低于前半段。

### 痛点 3：方向性错误代价最高

AI 最危险的失败模式不是"代码写错了"——那种错误测试能抓。最危险的是"方向选错了"：

- 需求理解偏差，做了三天发现不是用户要的
- 技术方案选择不当，做到一半发现行不通需要推倒重来
- 边界没定清楚，做着做着范围无限膨胀

这类问题的修复成本是实现错误的 10 倍以上。但在裸用 AI 的场景下，AI 收到一个模糊的需求后最自然的反应就是——立刻开始写代码。没有人（或规则）拦住它说"等一下，你确定理解对了吗？有没有考虑过替代方案？"

### 痛点 4：过程不透明，决策不可追溯

AI 对话是临时的。一个 feature 做完了，对话关闭，所有"为什么选方案 A 不选方案 B"、"为什么这个模块放在这里不放那里"的决策过程全部消失。

三个月后有人问"这段代码为什么这么写"，你除了 `git blame` 看到一个 commit message 外，什么上下文都没有。这对团队协作和长期维护是灾难性的。

## 三、思考过程：从痛点到设计原则

有了痛点清单，下一步是确定解决方向。这里有几个关键的思考转折点：

### 转折 1：约束对象是"过程"不是"产物"

最初的直觉是用 CI/CD 来兜底——写个 pre-commit hook 检查测试覆盖率，写个 linter 规则检查代码风格。但很快意识到这条路走不通：

**CI/CD 只能验证最终产物，无法约束工作过程。**

"先写测试再写实现"和"先写实现再补测试"的最终产物可能完全一样——都是一个通过的测试和一段工作的代码。但工程价值完全不同：前者证明了测试能捕获缺陷，后者只证明测试和代码恰好一致。

要约束的是 AI 的工作方式，不是工作结果。这意味着约束必须作用在 AI 的"行为指令"层面——也就是 CLAUDE.md。

### 转折 2：不是组合两个工具，而是设计一条流水线

市面上已经有解决单个痛点的能力模块：spec-driven workflow（结构化设计流程）、TDD 行为约束（强制验证）、subagent 分派（隔离上下文）。但这些能力模块各自独立存在时有一个致命问题：**没有人规定它们何时激活、以什么顺序协作。**

开发者可能在 TDD 模式下开始编码，完全忘记先做 proposal；或者写了完美的 spec 但实施时忘记激活验证约束。能力模块物理上共存，但逻辑上不协作。

SpecFuse 的核心工作不是"让你同时用 A 和 B"——而是**设计一条完整的流水线，让这些能力在正确的时机自动被调度**。用户感知到的不是"我在用 OpenSpec"或"我在用 Superpowers"，而是一个连贯的工作节奏：想清楚 → 审查一遍 → 动手做 → 验证归档。每一步背后调度了什么约束，用户不需要关心。

这就是"融合"和"组合"的区别。组合是"安装两个插件"，融合是"设计一条流水线让它们在正确的时刻自动接力"。

### 转折 3：声明式规则是当前唯一可行的方式

约束 AI 行为有两种路径：

1. **系统级强制**：修改 AI 工具链底层，让规则在运行时被强制执行（类似编译器的类型检查）
2. **声明级约束**：把规则写成自然语言指令，让 AI 在理解后自觉遵守

理想的选择当然是系统级强制——但现实是 Claude Code 没有暴露足够的 hook API。你无法在系统层面阻止 AI 跳过一个步骤。

所以 SpecFuse 选择了声明式——把规则写在 CLAUDE.md 中，利用 AI 的指令遵循能力来实现约束。**这是一个务实的选择，不是理想的选择。** 它的代价是约束没有 100% 的保障——在极端上下文压力下，AI 可能违反声明的规则。但在当前技术条件下，这是成本最低、效果最好的方案。

### 转折 4：规则需要分级——不是所有约束都一样硬

一开始尝试把所有规则都设为 MUST（必须执行），发现两个问题：

1. **过度约束会让 AI 在边缘场景做出荒谬的行为**。比如一个 3 行的 typo fix 也要创建 worktree 隔离——因为规则说"必须隔离"。
2. **所有规则都是最高优先级 = 没有优先级**。AI 在冲突场景下无法判断哪个规则该让步。

所以引入了 RFC 2119 风格的分级：

- **MUST**：无条件执行，没有例外。用于"永远正确"的规则——如"跑测试才能标完成"
- **SHOULD**：通常执行，但允许在给出理由后不执行。用于"通常正确但有例外"的规则——如"6+ 任务时启用 subagent"

SHOULD 的关键在于条件设得足够具体：不是"如果你觉得合适就启用 subagent"，而是"`tasks.md` 包含 ≥ 6 个独立任务 AND 上下文 > 40%"。具体的条件减少了 AI 主观解释的空间。

### 转折 5：流程成本必须自适应

如果给每个变更都套完整的 proposal → design → specs → tasks → grill → apply → verify → archive 流程，一个 5 分钟能解决的 bugfix 就变成了 30 分钟的文书工作。这不是约束，是拖累。

所以 SpecFuse 的流程复杂度随变更规模自动升级：

```
单行 bugfix     → TDD 铁律（最低约束）
中等变更        → TDD + 可选 worktree
大变更（≥6 任务）→ TDD + subagent + worktree（建议启用）
```

这背后的原则是：**流程成本是已知的线性开销，纪律缺失是未知的风险敞口。** 宁可多花 5 分钟流程时间，不赌一次"AI 说完成了但其实没测试"的未知风险。但这个"5 分钟"要和变更的风险成正比——不能一视同仁。

## 四、解决方案：SpecFuse 做了什么

基于以上思考，SpecFuse 的解决方案可以用一句话概括：

> **设计一条完整的 AI 工作流水线，把设计思考、质量审查、TDD 实施、验证归档编排成一个连贯节奏——用户只需按节奏推进，能力调度和纪律约束在背后自动接管。**

拆开来看：

### 方案 1：TDD 铁律——解决"完成声明不可信"

```
IRON LAW: NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE.

Before checking off a task [x], you MUST:
  a. Run the actual verification command in the terminal
  b. Paste the raw output as evidence
  c. Only then mark the task complete

"I believe it works" or "it should pass" is NEVER acceptable.
```

这条规则的设计是深思熟虑的：

- **"铁律"这个措辞不是修辞，是功能性的。** 它告诉 AI 这不是一条"建议"或"最佳实践"，而是一条不可协商的硬约束。AI 对指令强度措辞敏感——"should"和"MUST"、"recommendation"和"IRON LAW"在遵守率上有显著差异。
- **要求粘贴原始输出，不是要求"确认通过"。** 如果只要求 AI 回答"测试通过了吗？是/否"，它可能在没跑的情况下回答"是"。要求粘贴完整的终端输出，让验证行为变得可观察——人可以看到 AI 确实执行了命令。
- **"fresh"这个词防止缓存作弊。** 不接受"之前跑过一次"的证据，必须是当前状态下的新鲜执行。

### 方案 2：条件触发的 Subagent/Worktree——解决"大变更质量塌方"

```typescript
// 来自 src/templates/claude-md.ts renderApplyPhase()
`4. **Subagent-Driven Development trigger conditions:**
   When ALL of the following are met, SHOULD enable subagent mode:
   - \`tasks.md\` contains >= 6 pending implementation tasks
   - Most tasks have no \`blocked by\` dependency on each other
   - Tasks map to different modules/files with clear module boundaries
   - Session context window is >40% consumed`
```

这段声明解决了上下文饱和问题。但注意几个设计选择：

- **用 SHOULD 不用 MUST**：同样是 6 个任务，如果它们之间有紧密的数据依赖，分给独立 subagent 反而引入同步问题。SHOULD 留了判断空间。
- **四个条件用 AND 连接**：避免过早触发。只是任务多不够——还要任务独立、模块清晰、上下文确实紧张。
- **同时声明"不需要"的情况**：明确说 single-file bugfix、新增独立模块不需要 worktree，防止 AI 过度保守。

### 方案 3：强制设计阶段——解决"方向性错误"

SpecFuse 在实施之前插入一个不可跳过的设计阶段。当 `/opsx:apply` 被调用时，CLAUDE.md 要求 AI 必须先激活 brainstorming 技能：

- 一次只问一个问题（防止认知过载）
- 提出 2-3 种方案并比较取舍（防止锚定第一直觉）
- 明确 Non-goals（防止范围蔓延）

这不是"建议你先想想"——而是"没做完设计你就拿不到实施权限"。OpenSpec 的 artifact 依赖链强制了顺序：tasks 依赖 specs，specs 依赖 design，design 依赖 proposal。跳不过去。

### 方案 4：Artifact 链——解决"决策不可追溯"

每个变更自动产出 proposal → design → specs → tasks 四个文件。它们之间有引用关系：design 引用 proposal 的决策，specs 覆盖 design 声明的行为，tasks 追溯到 specs。

这不是文档审美——而是工程需要。三个月后有人问"为什么这样做"，答案就在 `openspec/changes/archive/` 里。

### 方案 5：CLAUDE.md 作为流水线调度协议

SpecFuse 的核心产物是一份 CLAUDE.md。但它不是一个"配置文件"——它是一份**调度协议**，声明了"哪个阶段激活哪些能力、以什么约束强度执行"。

```typescript
// 来自 src/templates/claude-md.ts renderApplyPhase()
`1. **MUST activate Superpowers \`test-driven-development\` as a pre-requisite skill.**
   ...
2. **MUST activate Superpowers \`verification-before-completion\` before marking ANY task done.**`
```

这段声明的作用不是"告诉 AI 你可以用 TDD"——而是**在流水线的特定节点自动激活 TDD 约束**。用户调用 `/opsx:apply` 时，不需要记得"哦我还需要手动开启 TDD"——协议已经把这件事编排好了。

用户在任何时刻只看到一条命令（`/opsx:propose`、`/opsx:apply`、`/grill-me`），背后的能力调度完全透明。这就是"融合"的用户体验：**感知到的是一个连贯节奏，不是多个工具的切换。**

为什么用 CLAUDE.md 而不是修改底层能力模块？

1. **调度和能力解耦**：底层能力模块各自迭代，SpecFuse 只负责"何时、以何强度激活它们"
2. **项目级调度策略**：不同项目可以有不同的调度配置——内部工具放松触发条件，金融系统收紧
3. **零侵入、可逆**：删除 CLAUDE.md 的 FUSION 块即回到无调度状态，底层能力不受影响

## 五、流程设计：四拍流水线

整体流程分为 **Think → Grill → Do → Verify** 四拍。前三拍对应软件工程的"设计 → 审查 → 实现 → 验证"——选择不创新是刻意的：工程师已经内化了这个心智模型，SpecFuse 的工作是把它编码为机器可执行的约束，而不是发明一种新的工作方式。

关键在于：用户不需要记住"现在该激活什么能力"。每一拍的命令（`/opsx:propose`、`/grill-me`、`/opsx:apply`、`/opsx:archive`）背后自动调度正确的约束——brainstorming、grill 协议、TDD 铁律、spec 校验——用户感知到的始终是一条连贯的工作节奏。

### Think 阶段（方案设计）

```
触发：/opsx:explore | /opsx:propose | /opsx:new
约束激活：Superpowers brainstorming
产出：proposal → design → specs → tasks
```

**做什么：** 在动手之前强制 AI 进入"思考模式"。不允许收到需求就开写。

**为什么这样设计：**
- 一次一问（苏格拉底式）→ 对应痛点 3 中"AI 直奔实现"的问题。一次抛 5 个问题用户会草率回答。
- 强制 2-3 方案对比 → 防止锚定效应。AI 想到的第一个方案不一定是最好的，但如果没人要求它想第二个，它就不会想。
- Non-goals 强制声明 → 边界不定清，范围必然蔓延。写下"这个变更不做什么"比写"做什么"更有约束力。

**两种节奏：**
- Path A（`/opsx:propose`）：用户已经想清楚了，一次性生成全部 artifact
- Path B（`/opsx:new` + `/opsx:continue`）：用户想逐步推进，一次一个 artifact，每步确认

这个分支的存在是为了适应不同的工作风格。有人喜欢一口气把设计做完，有人喜欢边做边想。SpecFuse 两种都支持，但两种都必须走完整个 artifact 链。

### Do 阶段（实施执行）

```
触发：/opsx:apply
约束激活：TDD + verification-before-completion + subagent/worktree（条件触发）
产出：通过测试的代码 + 验证证据
```

**做什么：** 按 tasks.md 逐个实施，严格 TDD，贴证据标完成。

**核心约束：**
1. 每个任务：写失败测试 → 最小实现 → 重构（Red-Green-Refactor）
2. 完成条件：跑测试 + 粘贴原始输出 + 标 [x]
3. 规模升级（条件触发）：任务多且独立 → subagent；破坏性变更 → worktree 隔离

**为什么 TDD 不是"建议"而是"铁律"：**

在 AI 编码场景下，TDD 的价值和传统开发不同。传统开发中 TDD 的主要价值是"驱动设计"——通过先写测试来思考接口。AI 编码中 TDD 的主要价值是**提供可验证的完成证据**——测试是你能观察到的、AI 确实做了验证这件事的证明。

如果测试是可选的，AI 在上下文压力大时会优先推进功能而跳过验证——这是观察到的实际行为，不是猜测。只有将其设为不可协商的规则，才能消除"这次跳一下应该没事"的滑坡。

### Verify 阶段（验收归档）

```
触发：/opsx:verify → /opsx:archive
约束：spec 逐条对照 + 全量测试 + linter 零警告
产出：归档记录
```

**做什么：** 确认实施与设计一致，没有遗漏或回归，然后归档。

**为什么需要单独的 Verify 阶段：**

Do 阶段的 TDD 验证了每个任务单独正确，但没有验证：
- 所有任务加起来是否覆盖了 spec 的全部要求
- 是否有回归（改了 A 模块，B 模块坏了）
- 代码质量基线是否维持

Verify 是全局视角的检查，补上 Do 阶段单任务验证的盲区。

### Grill 环节——Spec 的 Code Review

在 Think 完成后、Do 开始前，有一个 `/grill-me` 环节。它的定位是**给设计做 code review**：在代码还没写之前，对 artifact 做压力测试式审查。

**为什么在这个位置插入审查？**

传统 code review 发生在代码写完后。但 AI 编码的特点是：一旦开始实施，推翻设计的成本极高——AI 可能已经按照错误设计写了几百行代码和十几个测试。发现设计有问题时，沉没成本让人倾向于"凑合用"而不是"推翻重来"。

Grill 把审查前置到"还没写任何代码"的时刻：此时推翻设计的成本是修改几个文本文件，不是回滚几百行实现。

**审查协议设计：**

1. **备份机制**：审查开始前自动备份 change 目录到 `.grill-backup/`。如果审查发现重大问题需要回滚，执行 `grill-abort` 从备份恢复。成本是一次 `cp -r`，但保障了"审查怎么改都可以撤回"的安全感。

2. **一次一问**：每次只提出一个审查问题。和 brainstorming 的设计逻辑相同——密集轰炸导致用户仓促应付。每个问题包含：
   - 问题分级：`[blocking]`（必须解决才能继续）或 `[non-blocking]`（记录但不阻塞）
   - 具体发现描述
   - 推荐修改方案
   - 修改预览（dry-run diff）——**用户必须看到最终 diff 才能确认写入**

3. **审查维度**（按相关性选用，不是全查）：
   - 范围边界：goals/non-goals 是否清晰无歧义
   - 错误路径：异常、边界情况、失败场景是否覆盖
   - 依赖风险：新依赖评估，对现有模块的影响
   - 可测性：每条 spec 能否用当前测试基础设施验证
   - 安全性：输入验证、权限、数据暴露
   - 性能影响：O(n²)、阻塞 IO、内存压力
   - 向后兼容：是否有 breaking changes

4. **退出机制**：
   - `grill-stop`：退出审查，保留已做的修改
   - `grill-abort`：退出审查，从备份回滚所有修改
   - 模糊表达（"停吧"、"算了"）：AI 必须追问确认——防止误操作

5. **一致性扫描**：在审查总结前，自动执行跨 artifact 一致性检查：
   - 引用完整性：proposal 中的每个 goal 都映射到了 design/specs/tasks
   - 术语一致：同一概念全文使用同一名称
   - 逻辑冲突：没有 artifact 之间互相矛盾

**为什么 Grill 是"建议"而不是"强制"？**

强制审查对大变更有价值，但对小变更是过度开销。所以设计为：AI 在所有 artifact 完成后**主动推荐**执行 grill，但用户可以直接跳过进入 apply。这和 SHOULD 的设计逻辑一致——通常正确，但不是教条。

**Grill 在流水线中的位置：**

```
Think (brainstorming → artifacts) → Grill (spec review) → Do (TDD) → Verify (archive)
                                     ↑ 这里是设计缺陷成本最低的修复点
```

## 六、设计决策的代价与局限

每个设计决策都有代价。区别在于代价是否被识别、是否在可接受范围内。

### 代价 1：声明式约束没有系统级保障

CLAUDE.md 是纯文本。AI 在极端条件下可能违反规则。这是架构选择的根本代价。

**为什么接受这个代价：** 系统级强制需要 Claude Code 暴露 hook API，目前没有。声明式是唯一可行的方案。而且在实践中，AI 对 MUST 级别指令的遵守率足够高——不是 100%，但足以让流程有效运转。

**缓解方向：** 等 hook API 可用后，增加 post-action 验证层。当前用"粘贴证据"机制作为低成本替代。

### 代价 2：小变更的固定开销

修改一个错别字也要创建 proposal。这对于"5 分钟能解决"的问题是过度的。

**为什么当前接受：** v0.2 阶段确认 artifact 链路的完整性比优化小变更效率更重要。先保证流程跑通，再优化流程成本。

**缓解方向：** 计划引入 "quick-fix" 路径——满足特定条件（单文件、无 API 变更、有测试覆盖）时允许简化流程。

### 代价 3：仅支持 Claude Code

SpecFuse 依赖 CLAUDE.md 和 Superpowers 插件体系。Gemini CLI、Copilot 用户无法使用。

**为什么当前接受：** 融合规则的表达方式（CLAUDE.md + 技能激活协议）目前只有 Claude Code 原生支持。多平台适配需要理解每个平台对"行为约束"的支持深度——这不是简单的格式转换。

**缓解方向：** v0.3 路线图包含 Gemini CLI 适配。

### 代价 4：SHOULD 级别规则的实际效力低于预期

实践中观察到：AI 在满足 SHOULD 条件后，倾向于"继续做下去"而非"切换工作模式"。切换有认知成本，继续没有。

**缓解方向：** 考虑升级为"MUST + 显式 override"——默认强制执行，但 AI 可以声明具体理由后豁免。比 SHOULD 的单向建议更有约束力。

## 七、核心信念

回到最初的问题：SpecFuse 在赌什么？

赌的是：**在 AI 能力快速进步但可预测性尚未跟上的窗口期，结构化的流程约束是工程产出可靠性的必要条件。**

这个赌注的期限是有限的。当 AI 的行为可预测性足够成熟——当"AI 为什么跳过了测试"可以被精确归因和自动纠正时——SpecFuse 这种"提前声明规则然后希望 AI 遵守"的范式可能会让位给更轻量的"行为监控 + 实时纠正"模型。

但在那之前，SpecFuse 选择了保守的一边：流程成本是已知的、线性的、可控的；纪律缺失是未知的、非线性的、不可控的。

错了的代价是什么？多花一些流程时间。

不赌的代价是什么？每一次"AI 说完成了但实际没测试"都是一次盲赌。

工程的核心从来不是追求最优解，而是控制下行风险。SpecFuse 选了控制风险那一边。
